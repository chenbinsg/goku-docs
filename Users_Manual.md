## 1. 产品简介

AIOS（AI Operating System）是一款企业级 AI Agent 平台，提供以下核心能力：

- **智能对话**：与 AI Agent 进行多轮对话，Agent 可自主调用工具完成复杂任务
- **任务自动化**：将自然语言指令转化为可追踪的自动化任务
- **工作流编排**：通过可视化 DAG 编辑器编排多步骤、多分支业务流程
- **运营仪表盘**：实时展示系统资源、用户活跃、平台资产、任务统计与 LLM 费用（v1.1.0 新增）
- **三级组织架构**：支持「本部 → 部门 → Agent」三层组织结构（v1.1.0 新增）
- **企业通信集成**：全局配置 Email / 飞书 / Teams 通道，Agent 可按渠道推送通知（v1.1.0 新增）
- **多模型高可用**：集成 OpenAI、Anthropic Claude、OpenRouter（100+ 模型）、本地 Ollama 等主流模型，支持断路器自动故障转移（v1.2.0 新增 OpenRouter）
- **企业安全**：RBAC 权限控制、数据脱敏（DLP）、不可变审计日志、人工审批
- **可靠定时任务**：心跳引擎支持并发限流、失败重试、漏跑补偿和连续失败告警
- **来信自动处理**：客服、技术支持、IR 等 Agent 自主领取来信、理解意图、起草回复，经员工一键审批后自动发送（v1.9.3 新增）
- **Agent 访问控制**：四级可见性（公开 / 部门 / 角色 / 私有）+ 个人收藏（⭐），每位用户只看到有权使用的 Agent（v1.9.6 新增）
- **外部记忆源**：连接 Notion 工作区或 Obsidian Vault，笔记内容自动同步到知识库供 Agent 检索使用，每 30 分钟增量更新（v1.9.7 新增）
- **SSO 单点登录**：支持 GitHub / Google / 飞书 / 企业微信 / 钉钉 OAuth2 授权，无需单独注册（v1.9.7 新增）
- **部门主数据管理**：管理员预定义规范部门名，用户/Agent 的部门字段统一从候选列表选择，杜绝命名不一致（v1.9.6 新增）
- **用户自助改密**：所有用户均可在「个人中心」修改自己的密码，无需依赖管理员（v1.9.6 新增）
- **Microsoft 365 集成**：支持 Outlook 邮件读取、日历事件创建、会议室预订
- **三语支持**：界面支持中文、英文、日文切换；工作流编辑器、认知中枢均完整支持三语
- **命令行工具（CLI）**：`aios-cli` 支持 JWT 认证、实时流式输出、对话、任务管理（v1.2.0 全面重写）
- **消息表情反应**：对 Agent 回复点击 emoji 表达态度（👍👎❤️😂😮🎉），反应持久保存（v1.2.7 新增）
- **@Agent 精准路由**：单条消息发送时可选择/输入 `@AgentName` 覆盖会话默认 Agent（v1.2.7 新增）
- **模型能力自适应**：系统自动检测当前模型的能力（扩展思考、视觉识别、API 参数），切换模型无需手动调整（v1.2.7 新增）
- **行业技能包**：日本 IR 助理（26 工具）与市场情报（4 工具）开箱即用（v1.2.0 新增）
- **组织架构 API**：开放 REST API 支持批量同步 Agent 组织归属，兼容 HR 系统 CSV 导出格式（v1.2.0 新增）
- **工作区可读目录**：Agent 工作区目录以用户名命名（如 `alice/`），替代原来的 UUID，方便管理员直接在文件系统中定位用户文件（v1.2.1 新增）
- **并发实例管理（Project Wukong）**：每种 Agent 类型拥有独立的并发槽位池，超出时请求自动排队等待。代理管理页面实时显示各 Agent 的负载，管理员可在运行时动态调整并发上限（v1.2.2 新增）
- **三级组织架构（Teams）**：在「部门 → 团队 → 用户」三级体系下，`组织架构` 页面（`/org`）以树形视图展示全貌，管理员可一站式管理部门、团队及成员（v1.9.8 新增）
- **DefaultDeny 访问策略**：Agent 默认拒绝所有非创建者访问，管理员通过「访问策略」Tab 按用户 / 团队 / 部门 / 租户颗粒度精细授权（v1.9.8 新增）
- **分析看板增强**：DAU/WAU/MAU 分段切换、Cohort 留存热图、部门/团队维度过滤（v1.9.8 新增）

## 3. 账号与设置

### 个人中心（v1.9.6 新增）

路径：右上角头像 → **个人中心**（`/profile`）

个人中心页面展示当前用户的基本信息，并提供自助改密功能：

| 信息项 | 说明 |
|--------|------|
| 用户名 | 账号登录名（只读） |
| 邮箱 | 联系邮箱（只读，如需修改联系管理员） |
| 身份 | Admin / 普通用户标签 |
| 部门 | 所属部门 |
| 状态 | 账号是否启用 |

**修改密码**

在「修改密码」卡片中输入当前密码和新密码（至少 6 位）即可完成修改，无需联系管理员。

### 多因素认证（MFA）

账号设置页面可开启 TOTP（Google Authenticator 等应用）二次验证，提升账号安全性。

### 界面语言

点击右上角语言切换器，即时切换中文 / English / 日本語，无需刷新页面。城市时钟的城市名和日期格式也会随语言同步切换。

## 5. 智能对话（Chat）

Chat 是与 AI Agent 交互的主界面，登录后默认进入此页面。

### 5.1 发起对话

1. 在左侧侧边栏点击 **「+ 新对话」**
2. 在底部输入框中输入消息，按 **Enter** 换行，**Cmd+Enter**（macOS）或 **Ctrl+Enter** 发送

### 5.2 工具栏说明

| 图标 | 功能 |
|------|------|
| 📎 附件 | 上传文件（图片、文档等）供 Agent 处理 |
| 中 / 日 / EN | 语音输入语言选择 |
| 🎤 麦克风 | 语音输入（需在系统设置中授权麦克风） |
| 📍 位置 | 附加当前地理位置信息 |
| 🔊 TTS | 开启/关闭文字转语音朗读回复 |
| Agent 选择 | 选择本次对话默认使用的 Agent（通用 / 专属） |
| 默认模型 | 临时覆盖本次对话使用的 LLM 模型 |
| Medium ⚡ | 调整思考深度（Low / Medium / High / Max）；**仅在当前模型支持扩展思考时显示**（v1.2.7 动态判断） |
| 紧急重置 | 强制终止当前卡住的任务，重置发送状态 |

### 5.2a @Agent 精准路由（v1.2.7 新增）

发送消息时可临时切换执行 Agent，无需修改对话设置：

**方式一：下拉选择器**  
输入框上方出现「Route to agent…」下拉列表，从中选择目标 Agent 后发送，消息仅由该 Agent 处理。发送后自动重置为默认 Agent。

**方式二：@提及输入**  
在输入框中输入 `@` 后跟 Agent 名称，系统弹出自动补全浮层，选中后形成蓝色 Chip 标签；发送即路由到该 Agent。

> 每条消息的路由独立，不影响对话整体设置。

### 5.2b 消息表情反应（v1.2.7 新增）

鼠标悬停在 Agent 回复气泡上，底部出现 😊 触发按钮，点击展开 6 个 emoji 选择器：

| Emoji | 含义 |
|-------|------|
| 👍 | 有帮助 |
| 👎 | 没帮助 |
| ❤️ | 喜欢 |
| 😂 | 有趣 |
| 😮 | 惊讶 |
| 🎉 | 棒极了 |

再次点击同一 emoji 可取消反应。反应数据保存在浏览器本地，刷新页面后保留。

### 5.3 斜杠命令

在输入框中输入 `/` 会弹出命令菜单：

| 命令 | 功能 |
|------|------|
| `/help` | 查看帮助信息 |
| `/think` | 开启深度思考模式 |
| `/model` | 切换对话模型 |
| `/status` | 查看当前任务状态 |
| `/clear` | 清空当前对话上下文 |

### 5.4 富文本卡片

Agent 回复可能包含以下交互卡片：

| 卡片类型 | 说明 |
|---------|------|
| **审批卡片** | 需要人工确认的高风险操作，点击「批准」或「拒绝」 |
| **代码卡片** | 带语法高亮的代码块，支持一键复制 |
| **表格卡片** | 结构化数据展示，支持搜索和导出 |
| **表单卡片** | Agent 请求用户填写结构化数据 |
| **文章卡片** | 长文内容展示，支持 Markdown 渲染 |

### 5.5 工具调用追踪

Agent 执行期间，对话区会显示正在调用的工具名称和执行状态，方便追踪 Agent 的思考过程。

### 5.6 人工审批拦截

当 Agent 准备执行高风险操作（如删除文件、发送邮件、执行代码等）时，会自动暂停并弹出审批卡片，需人工点击「批准」后才会继续执行。

## 6. 任务管理

路径：左侧菜单 → **任务管理**

### 6.1 任务列表

展示所有 Agent 任务，含以下信息：
- 任务 ID 与标题
- 状态：`pending`（等待）/ `running`（执行中）/ `completed`（完成）/ `failed`（失败）/ `cancelled`（已取消）
- 创建时间、完成时间
- 关联的 Agent 类型

支持按状态筛选、关键词搜索。

### 6.2 任务详情

点击任务行查看：
- **执行步骤**：每一步的工具调用、输入输出
- **思考过程**：Agent 的推理链（Thinking）
- **最终结果**：格式化输出

### 6.3 手动创建任务

点击右上角 **「+ 创建任务」**，填写：
- 任务提示词（自然语言描述）

## 7. 工作流

路径：左侧菜单 → **工作流**

### 7.1 工作流列表

- 触发立即执行
- 编辑、删除工作流

### 7.2 创建/编辑工作流

工作流使用**可视化 DAG 编辑器**（基于 ReactFlow）：

**节点类型**：

| 类型 | 颜色 | 说明 |
|------|------|------|
| 开始节点 | 绿色 | 工作流触发入口 |
| Agent 节点 | 蓝色 | 执行一个 Agent 任务 |
| 条件节点 | 橙色 | 根据前序节点结果进行分支判断 |
| 并行节点 | 黄色 | 同时执行多个子任务 |
| **HTTP 请求节点** | 青色 | 调用外部 REST API（v1.2.7 新增，见下） |
| 结束节点 | 红色 | 工作流结束，输出最终结果 |

**HTTP 请求节点配置**（v1.2.7 新增）：

| 字段 | 说明 |
|------|------|
| 方法 | GET / POST / PUT / DELETE |
| URL | 目标接口地址（支持 `{{变量}}` 插值） |
| 请求头 | JSON 格式，如 `{"Authorization": "Bearer xxx"}` |
| 请求体 | JSON 格式，POST/PUT 时使用 |
| 超时（秒） | 默认 30 秒 |
| 输出变量名 | 响应体存入此变量，后续节点可引用 |

响应结果包含 `status_code`、`ok`（布尔）、`body` 三个字段，可在条件节点中使用。

**操作**：
- **拖拽**：从左侧调色板拖动节点到画布指定位置（v1.2.7 优化精确坐标）
- **点击 +**：点击调色板节点右侧 **+** 按钮，节点添加至画布中心
- 连线表示执行顺序，点击节点配置参数
- 右上角保存

**界面语言**：工作流编辑器完整支持中文、英文、日文（v1.2.7 新增）。

### 7.3 工作流监控

执行中的工作流实时显示每个节点的状态（等待 / 执行中 / 完成 / 失败），节点颜色区分状态。支持从中断点重试。

|--------|------|---------|

**多模型故障转移**

### 9.7 长期记忆

路径：AI 能力 → **记忆**

管理跨会话的持久化记忆：

- **查看记忆列表**：Agent 自动积累的事实、偏好、上下文
- **删除记忆**：移除不再需要的条目
- **记忆类型**：事实（fact）/ 偏好（preference）/ 上下文（context）

Agent 在每次对话中会自动检索相关记忆注入上下文。

**用户偏好自动提取（v1.2.0 新增）**：

任务完成后，系统自动从执行步骤中提取用户偏好（如语言风格、报告格式、输出语言等），以标签 `__pref__` 存储到记忆库，并在后续对话中自动注入。

| 配置项 | 环境变量 | 默认值 | 说明 |
|--------|----------|--------|------|
| 启用提取 | `AIOS_PREF_EXTRACT` | `true` | 是否开启偏好自动提取 |
| 最小步数 | `AIOS_PREF_MIN_STEPS` | `2` | 任务至少执行几步才触发提取 |
| 每日上限 | `AIOS_PREF_MAX_DAILY` | `30` | 每天最多提取多少条偏好 |
| 注入条数 | `AIOS_PREF_INJECT_TOP_K` | `5` | 每次对话注入最多几条偏好 |

> 偏好记忆**永不过期**，但可在记忆管理页面手动删除。API 支持通过 `?tag=__pref__` 筛选偏好记忆：`GET /api/v1/memory?tag=__pref__`

### 9.9 外部记忆源

> **v1.9.7 新增**

路径：AI 能力 → **外部记忆源**（`/knowledge/external-sources`）

将 Notion 工作区或本地 Obsidian Vault 的笔记内容自动同步到 AIOS 知识库，Agent 执行任务时可透明检索这些内容，无需手动上传。

#### 连接 Notion

1. 在 [Notion Developers](https://www.notion.so/my-integrations) 创建公开集成（Public Integration），获取 `Client ID` 和 `Client Secret`
2. 在 AIOS `.env` 文件中配置：
   ```ini
   NOTION_CLIENT_ID=your-client-id
   NOTION_CLIENT_SECRET=your-client-secret
   ```
3. 重启 AIOS 后端
4. 进入 **外部记忆源** 页面 → 点击「添加数据源」→ 选择 **Notion** 标签 → 点击「连接 Notion」
5. 在弹出的 Notion OAuth 授权页面授权后，页面自动关闭并开始同步

**同步规则：** 所有在 Notion 集成中被共享的页面均会被同步；新增或修改页面后，下次同步（最多 30 分钟）将自动更新到知识库。

#### 连接 Obsidian

Obsidian Vault 需在 AIOS 服务器本机上（本地部署）或服务器可访问的挂载路径上：

1. 进入 **外部记忆源** 页面 → 点击「添加数据源」→ 选择 **Obsidian** 标签
2. 填写 Vault 目录的绝对路径，例如 `/home/user/Documents/MyVault`
3. 可选填写显示名称，点击「添加数据源」

**同步规则：** 系统扫描 Vault 内所有 `.md` 文件，识别 YAML Frontmatter（`title`、`tags`），仅同步自上次扫描后有变更的文件（基于 `mtime`）。

#### 数据源状态说明

| 状态 | 说明 |
|------|------|
| 正常（绿色） | 上次同步成功，等待下次定时同步 |
| 同步中（蓝色旋转） | 正在获取并索引内容 |
| 错误（红色） | 最近一次同步失败，悬停可查看错误信息 |
| 已断开（灰色） | 数据源已手动禁用 |

#### 操作

| 操作 | 说明 |
|------|------|
| 立即同步 | 不等定时，立即触发一次全量/增量同步 |
| 删除 | 移除数据源，并删除该数据源产生的所有知识库文档 |

## 10. 扩展与集成

### 10.1 MCP 服务器

路径：**MCP 服务器**

管理 Model Context Protocol 服务器 —— 接入外部或内置的 MCP 服务器，把它们
的能力（capabilities）纳入平台，经授权后供 AI Tool / Agent 调用。

> 技术细节（接入模型、配置规范、授权机制、内置服务器开发）见
> 《AIOS 综合技术标准》Part B（MCP 服务器接入标准）。本节是面向操作的使用说明。

#### 快速上手：5 步接入一个 MCP 服务

不管接的是 npx 包、远程 HTTP 服务还是内置 server，端到端都是这 5 步。
先按这条主线走通，再看后面各小节的细节。

1. **新增服务器**（MCP 管理 → MCP 服务器 →「新增服务器」）：选预置模板，
   或手动填连接方式 + 启动命令 / 端点。
2. **配凭证**（按需）：若服务需要外部系统密钥，先在「外部连接管理」建一条
   对应类型的连接（§10.1a），回到服务器表单里绑定；零凭证的服务跳过这步。
3. **测试连接**：点「测试连接」做一次真实探测，健康状态变绿才算通。
4. **同步能力**：点「同步能力」，把服务器的 tools / resources / prompts
   拉进平台。同步后「能力」Tab 能看到具体能力。
5. **授权调用方**：在「能力」Tab 点目标能力的「**管理**」，在抽屉里把它授权给某个
   **Agent** 或 **AI Tool**（默认拒绝，不授权调不动）。之后把对话路由到该 Agent 即可使用。

> 没走到第 5 步、或对话没指定具体 Agent，调用会直接返回
> `MCP_CAPABILITY_NOT_AUTHORIZED`。这是最常见的"接好了却用不了"原因。

#### 列表页

顶部 4 个统计卡：服务器总数 / 运行中 / 异常 / 已停用。下方是服务器列表，
支持按名称编码、服务分类、状态、健康状态筛选。

#### 新增服务器

点「**新增服务器**」，抽屉有两个 Tab：

- **预置**：选一个内置模板（GitHub / Filesystem / Memory / SQLite /
  PostgreSQL / S3 存储 等），表单自动预填启动命令与环境变量键。选中后
  补齐凭证类空值即可。
- **手动**：完全自定义。关键字段：
  - **连接方式**：`stdio`（子进程）/ `http`（Streamable HTTP）
  - **启动命令**（stdio）：如 `npx -y @modelcontextprotocol/server-github`
  - **env_config**：子进程环境变量（JSON）。凭证（token / AWS key 等）
    填这里，保存时**自动加密**入库，页面与接口都不会回显。

#### 服务器详情页（6 个 Tab）

页面顶部是概要卡（名称 / 编码 / 分类 / 连接 / 状态 / 健康 + 能力计数 +
最近探测时间）和 4 个操作按钮（测试连接 / 同步能力 / 编辑 / 启停）。下方 6 个 Tab：

| Tab | 内容 |
|---|---|
| 基本信息 | 名称、编码、分类、负责人、状态 |
| 连接配置 | 连接方式、启动命令 / 端点、认证、env_config |
| 能力 | 同步来的能力 / Resources / Prompts；每行可「**测试调用**」，也可点「**管理**」打开该能力的授权抽屉（见下）|
| 健康监控 | 当前健康状态 + 探测记录时间序列 |
| 调用日志 | 能力调用记录（含授权 / 配额归因、错误码）|
| 变更记录 | 服务器 / 能力 / 授权的审计变更 |

> ⚠️ **没有独立的「授权调用方」Tab 了**。授权是**按单个能力**管理的——在
> 「能力」Tab 点某能力行的「管理」，在抽屉里完成（见《授权调用方》小节）。

#### 测试连接 与 同步能力

- **测试连接**：平台拉起 / 连接服务器做一次探测，刷新健康状态。
- **同步能力**：拉取服务器的 tools / resources / prompts 写入平台。
  同步完成后「能力」Tab 表格自动刷新。上游已移除的能力会标为
  `inactive`（不删除，保留历史调用日志可解析）。

#### 授权调用方（默认拒绝，按能力管理）

MCP 能力**默认不可被任何调用方调用**。授权**不是服务器级、而是能力级**：
在「能力」Tab 找到目标能力 → 点该行「**管理**」→ 抽屉里有 4 段（可折叠）：

| 抽屉段 | 作用 |
|---|---|
| **能力详情** | 能力名 / 说明 / 状态 / 入参 / 出参 schema / 最近同步与调用时间 |
| **授权模式** | 在「需要授权（默认）↔ 公开」之间切换；公开模式下可维护黑名单 |
| **限额** | 给这个能力设**总配额**（周期 + 上限），下方实时显示"总额 / 已分配 / 可分配余量" |
| **授权调用方** | 给具体 Agent / AI Tool 授权，可选填**单授权配额**（从总配额里切一份）；每条授权可启用 / 停用 / 删除 |

操作顺序：先在「限额」段（按需）给能力设总配额 → 在「授权调用方」段点
「新增授权」选 `Agent` / `AI Tool` + 调用方 + 单授权配额 → 保存即生效。
所有变更进「变更记录」Tab。

调用未授权的能力会返回结构化错误：`MCP_CAPABILITY_NOT_AUTHORIZED` /
`MCP_AUTHORIZATION_DISABLED` / `MCP_AUTHORIZED_QUOTA_EXCEEDED` /
`MCP_CAPABILITY_QUOTA_EXCEEDED`。

> 早期版本通过编辑 `workspace/.mcp.json` 配置服务器；现在统一在本页面
> 管理，配置存数据库、密文加密，无需改文件、无需重启后端。

#### 能接入哪些服务：纯页面配置 vs 需要开发

本页面**没有固定的"支持服务白名单"**——`启动命令` 是自由文本，理论上能拉起
任意 npx / uvx 包，`http` 连接方式能对接任意暴露了 MCP 端点的远程服务。
一个服务能不能**只靠页面配置**接入，取决于它**怎么拿到凭证**。按下面三档判断：

**第 ① 档 · 零配置，选了就能跑**（不需要任何外部凭证）

| 预置模板 | 接入的服务 |
|---|---|
| Filesystem | 本地目录读写 |
| Fetch | 抓取网页 / HTTP |
| Memory | 知识图谱式记忆 |
| Time | 时间 / 时区 |
| Puppeteer | 无头浏览器自动化（Chromium 依赖已随镜像安装）|
| SQLite | 本地 SQLite 文件 |

**第 ② 档 · 配一份凭证就能用**（仍然不碰代码）

| 预置模板 | 接入的服务 | 凭证怎么给 |
|---|---|---|
| GitHub | GitHub | 「外部连接管理」建一条 `github` 连接后绑定（见 §10.1a）|
| Slack | Slack | 建一条 `slack` 连接后绑定 |
| PostgreSQL | Postgres | 建一条 `database` 连接后绑定 |
| Brave Search | Brave 搜索 | 直接在 `env_config` 填 `BRAVE_API_KEY`（带厂商前缀的专用键允许直填）|
| S3 存储 | S3 / S3 兼容对象存储 | 建一条 `s3` 连接后绑定（内置 server，已就绪）|
| MySQL | MySQL | 建一条 `database` 连接后绑定（内置 server，已就绪）|

**判断口诀**：服务需要的密钥若是**厂商专用名**（如 `BRAVE_API_KEY`、
`NOTION_API_KEY`、`GITLAB_TOKEN`），直接填 `env_config` 即可（保存自动加密）；
若它需要的是被「外部连接」接管的**通用 / 占位名**（`AWS_*`、`GITHUB_TOKEN`、
`SLACK_BOT_TOKEN`、裸 `TOKEN` / `API_KEY` / `PASSWORD` 等），`env_config`
会拒绝保存（错误码 `MCP_ENV_SECRET_FIELD_NOT_ALLOWED`），必须改走「外部连接」
——而外部连接目前只支持 7 种类型（s3 / sftp / url / local_path / database /
github / slack）。

**第 ③ 档 · 必须先开发**（找研发，不能自助）

- 某个服务的密钥落在「外部连接」类型之外，且不是厂商前缀键 → 需要新增一个
  连接类型 + 注入分支（见技术标准《十九、外部连接管理》）。
- 没有现成 npx 包、也没有 HTTP MCP 端点能跟它通话，需要平台自己写一个内置
  Python server。**典型例子：ClickHouse**——它虽属 `database` 类，但通用
  数据库 server 当时不支持其方言，研发新增了 `clickhouse_server.py` 才能用。

**两个生效前提**（满足不了，①② 档也会失败）：

1. **运行环境要能访问 npm**：`npx -y` 是**首次启动时按需下载**包的，内网隔离 /
   无公网出口的部署会在拉起子进程时失败。生产镜像已内置 Node 20 + npx。
2. **包名要真实存在**：`@modelcontextprotocol/server-*` 系列随上游演进有改名 /
   归档，填之前请在 npm 上确认该包仍按此名发布。

#### 接入远程 HTTP MCP 服务（不拉子进程）

除了 stdio（本地拉起子进程），平台还能直接对接**已经部署在别处、对外暴露了
MCP HTTP 端点**的服务——不需要 npm、不在本机起进程，最省事。平台里
`aibi-qa`、`opensearch-log-search` 就是这么接的。

新增服务器时：

1. **连接方式**选 `http`（Streamable HTTP）。这是**传输协议**，在 MCP 服务器表单里选，
   不是去「外部连接管理」里选——别和外部连接的类型搞混（见下方「两个 HTTP，别混」）。
2. 填 **服务端点 URL**（对方的 MCP HTTP 地址）。
3. 鉴权（端点公开则跳过）：若 Goku 调用对端需要带 `Authorization` 头，去
   「外部连接管理」建一条 **`HTTP 接口`（`url` 类型）** 连接，在 secret 里填
   `authorization_header`（如 `Bearer xxx`）；回服务器表单把它绑到
   **「MCP Server 调用鉴权」** 槽。运行时平台据此注入 `Authorization` 头，
   **不写进 `env_config`**。
4. 之后同样走「测试连接 → 同步能力 → 授权」三步。

> **两个 HTTP，别混**：
> - **传输是不是 HTTP** → 在 MCP 服务器表单的「连接方式」选 `http`（对应情形 A）。
> - **HTTP 端点要不要鉴权** → 在「外部连接管理」建 `HTTP 接口`（`url`）连接，绑到
>   「MCP Server 调用鉴权」槽（对应情形 B）。
>
> 端点无需鉴权时，只做第 1、2 步即可，一条外部连接都不用建。

> 适用场景：你们已有一个独立部署的 MCP 服务（自研或第三方 SaaS 暴露的 MCP
> 端点），只想让本平台消费它，而不想把它的运行托管到本平台。

#### 内置服务器（无需 npm，python -m 启动）

平台自带数个内置 MCP 服务器（代码在 `backend/app/agent/mcp/servers/`，
以 `${VENV_PYTHON} -m` 形式启动，无需安装第三方包），部分可在「预置」里直接选用：

| 服务器模块 | 功能 | 是否需外部连接 |
|---|---|---|
| `file_parser_server` | 文件解析（PDF / Excel / 图片 OCR 等），不连任何外部系统 | 否 |
| `storage_s3_server` | S3 / S3 兼容对象存储（presigned 上传 / 下载、元数据、删除）| 是（`s3`）|
| `db_query_server` | 通用数据库查询（MySQL 等）| 是（`database`）|
| `reconciliation_db_server` | 商户对账独立库（读写）| 是（`database`）|
| `clickhouse_server` | ClickHouse 查询 | 是（`database`）|
| `bank_recon_server` | 银行对账数据源 | 否（读本地数据目录）|
| `flight_travel_server` | 航班 / 差旅 | 否 |
| `ira_server` | IRA 数据源 | 否 |

> 新增一个内置 server 属于研发动作（《MCP 服务器接入技术标准》十六章），
> 不是页面自助操作。

#### Agent / AI Tool 如何使用 MCP 能力

平台对所有 MCP 能力调用强制 **默认拒绝 + 显式授权 + 配额**。两条调用路径,
都走同一套授权检查链:

**路径 A：Agent 调用(隐式,LLM 自己决定)**

启动时,所有 active MCP 能力会自动注册成普通 Agent 工具,名字格式：

```
{服务器编码}__{能力名}
```

例：`github__create_issue`、`storage-s3__create_presigned_upload_url`、
`memory__search_nodes`。

Agent 在 ReAct 循环中通过工具调用机制使用它们 —— Agent 看不到 "MCP"
这个概念,就是个普通工具。**要真正调起来,需同时满足三个条件**：

1. **对话已路由到具体 Agent**（`/agent/...` 路径、`@mention`、或前端
   「指定 Agent」按钮）——默认对话调不动(见下方 ⚠️)。
2. **该 Agent 在对应能力的「管理」抽屉里被授权**——默认拒绝,见《授权调用方》。
3. **该 MCP 工具在 Agent 的工具池里**——MCP 能力会自动进默认工具池;但**如果
   这个 Agent 配置了 `allowed_tools` 白名单,必须把 `{服务器编码}__{能力名}`
   也加进白名单**,否则即便授权了,Agent 也"看不见"这个工具、自然调不动。
   这是"授权了却没反应"的常见原因之一。

> 🔔 **MCP 工具默认需要人工审批**。MCP 能力的权限级别是 `2`(approval),
> Agent 选用某个 MCP 工具时,执行器会**先暂停、发起审批**(聊天里弹
> `⚠️ Agent 需要您的授权才能继续` + 推送到 [审批中心](/approvals)),
> 批准后才真正调用,拒绝 / 超时则中止(fail-closed)。内置与外部 MCP 服务
> **一视同仁**,都要审批。这不是故障——看到 Agent"卡住等确认"是正常行为。
> 审批超时由 `AGENT_APPROVAL_TIMEOUT` 控制。

调用被拒时, LLM 会看到结构化错误,可以读懂并决定如何处理:

```json
{"error": "...", "error_code": "MCP_CAPABILITY_NOT_AUTHORIZED"}
```

错误码:`MCP_CAPABILITY_NOT_AUTHORIZED` / `MCP_AUTHORIZATION_DISABLED` /
`MCP_AUTHORIZED_QUOTA_EXCEEDED` / `MCP_CAPABILITY_QUOTA_EXCEEDED`。

> ⚠️ **默认对话（没指定具体 Agent）调 MCP 工具会立刻返回
> `MCP_CAPABILITY_NOT_AUTHORIZED`**。要让某个聊天能用 MCP,必须先路由到
> 一个具体的 Agent,再给该 Agent 显式授权。

**授权模式按能力切换(可选公开 + 黑名单)**

每个能力可在其「管理」抽屉的「**授权模式**」段里切换:

- **需要授权(默认)** —— 默认拒绝,只有「授权调用方」段里显式授权的
  Agent / AI Tool 能调用
- **公开** —— 全部可用,无需授权;可在弹窗里维护**黑名单**,黑名单里
  的 (调用方类型, 调用方) 命中即拒

切到「公开」不会删掉已有授权;切回「需要授权」时它们自动重新生效。

**路径 B：AI Tool 调用(显式,外部 / 编排层)**

外部系统或编排层用 HTTP 直接调:

```http
POST /api/v1/ai-tools/{tool_id}/invoke
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "mcp_capability_id": "<能力 UUID>",
  "input": {
    "key_prefix": "uploads",
    "filename": "report.pdf",
    "content_type": "application/pdf"
  },
  "session_id": "optional-session-uuid",
  "invoke_type": "agent_auto"
}
```

成功返回 `{call_log_id, result: "success", response_time_ms, output_summary}`。
失败 HTTP 403,`detail` 含结构化错误码与原因。

**怎么给 Agent / AI Tool 授权**

1. MCP 管理页 → 选目标服务器 →「**能力**」Tab
2. 找到要授权的能力,点该行「**管理**」打开抽屉
3. （可选）「**限额**」段给该能力设总配额(周期 + 上限)
4. 「**授权调用方**」段点「**新增授权**」:
   - 调用方类型: `AI Tool` 或 `Agent`
   - 调用方: 从下拉里选具体的 AI Tool / Agent
   - **单授权配额**(选填): 从该能力总配额里切一份给这个调用方
5. 授权可启用 / 停用 / 编辑额度 / 删除,所有变更都进「变更记录」Tab

> 抽屉锁定的就是这一个能力,所以不需要再选"目标能力"——要授权另一个能力,
> 回「能力」Tab 点那一行的「管理」。

### 10.1a MCP 外部连接(v1.2.x 新增)

路径:**MCP 管理 → 外部连接管理**

把分散在多个 MCP Server 里的连接配置 + 密钥集中托管,一份配置可被
多个 MCP Server 共用,密钥统一加密、统一审计。

> 这里不是新的 MCP Server,也不影响现有 MCP Server 的授权边界。
> 多套需要独立授权的 S3 / SFTP 仍然各开一个 MCP Server 实例。

**新建一条外部连接**

1. 进入「外部连接管理」→「新增连接」
2. 选择连接类型(`s3` / `sftp` / `url` / `local_path` / `database` /
   `github` / `slack`),表单会自动填入该类型的模板
3. 填写:
   - **编码**:业务唯一标识,MCP Server 引用时用这个值
   - **名称**:展示用
   - **配置(config)**:非敏感字段(endpoint / region / host / 默认
     bucket 等)
   - **密钥(secret)**:敏感字段;保存时全部加密,API 返回时显示
     为 `已配置 ********`
   - **允许范围(allowed_scopes)**:安全闸,**必填**。运行时所有访问
     都会按此范围检查:
     - `s3`:`allowed_buckets` / `allowed_prefixes`
     - `sftp`:`allowed_paths`
     - `url`:`allowed_domains`(同时强制 http(s)、自动拦截私网 IP)
     - `local_path`:`allowed_dirs`(自动拦截 `/etc /root /proc /sys`
       等系统目录)
     - `database` / `github` / `slack`:`allowed_tables` / `allowed_repos`
       / `allowed_channels`
4. 保存后可点「测试」做一次真实探测,状态会更新为 success / failed,
   并显示最近一次错误

**编辑密钥的行为**

打开已有连接时,所有密钥字段显示为 `已配置 ********`。
- 不动它 → 保存时后端识别为"未修改",保留原密文,不会重新加密
- 把它改成新值 → 保存时按新值重新加密

**MCP Server 怎么绑定外部连接(v1.2.x 端到端打通)**

1. **MCP 管理 → MCP 服务器**,新增 / 编辑要用 S3 / GitHub / Slack /
   PostgreSQL 等外部系统的 server
2. 表单中段会出现 **「外部连接」下拉** —— 选你在「外部连接管理」
   里创建好的对应连接(下拉只列 enabled 状态的);
   storage-s3 / server-github / server-slack / server-postgres
   这几个 preset 是**必选**,没选保存不通过
3. **env_config(JSON dict)只用来存本实例运行参数**(TTL / 超时 /
   限额等)。**不允许**再往里写 `AWS_ACCESS_KEY_ID` / `GITHUB_TOKEN` /
   `DB_PASSWORD` 这类外部 secret(后端会直接拒绝并报
   `MCP_ENV_SECRET_FIELD_NOT_ALLOWED`),旧 MCP Server 还在用这种
   写法的,必须把 secret 迁到外部连接、把这些字段从 env_config 删掉

**保存与启动时会发生什么**

- 保存:后端校验绑定的连接存在、enabled、type 与 server 类型匹配,
  失败立刻报 400 + 结构化错误码
- 启动 / 测试连接 / 能力同步:runtime 自动按外部连接的类型注入对应
  env vars(AWS_*、SFTP_*、DB_*、URL_*、LOCAL_ALLOWED_DIRS、
  GITHUB_*、SLACK_* …),MCP Server 子进程拿到的环境干净,
  `connection_id` 本身不会传给子进程

**两个绑定槽:数据连接 vs 调用鉴权(别搞混)**

一个 MCP Server 表单上其实有**两个**外部连接绑定槽,用途完全不同:

| 槽 | 字段 | 回答的问题 | 接受类型 |
|---|---|---|---|
| **数据连接** | `connection_id` | 这个 server **去访问的外部系统**用什么凭证(S3 桶、数据库、GitHub…)| s3 / sftp / database / github / slack / url / local_path |
| **MCP Server 调用鉴权** | `server_auth_connection_id` | **Goku 调用这个 server 端点本身**要不要带鉴权头 | **只接受 `HTTP 接口`(`url`)** |

- **stdio 本地 server**(storage-s3 / github…):只用「数据连接」槽,把凭证注入子进程环境。
- **远程 `http` server**:端点要鉴权时用「MCP Server 调用鉴权」槽,平台据此给请求加
  `Authorization` 头;选别的类型会报 `MCP_CONNECTION_TYPE_UNSUPPORTED`。

**`HTTP 接口`(`url`)类型的两种用途**

`url` 类型(下拉里显示为「**HTTP 接口**」)同时服务两个场景,所以它带
`authorization_header` + `allowed_domains` 两个关键字段:

1. **file-parser 从 HTTP URL 下载文件**——绑到 file-parser server 的「数据连接」槽,
   `allowed_domains` 限定可下载域名(强制 http(s)、自动拦私网 IP)。
2. **给远程 HTTP MCP Server 端点做调用鉴权**——绑到「MCP Server 调用鉴权」槽,
   只借用其中的 `authorization_header`。

> 历史包袱:这个类型早期只做用途 1,所以一度叫"URL 文件源";现已统一显示为
> 「HTTP 接口」。看到旧截图里的"URL 文件源"指的就是它。

**多 S3 怎么分别授权**

- 准则:外部连接是"连接配置 + 密钥",不是授权边界
- 需要给不同调用方分别授权的多套 S3 → 创建多个 MCP Server 实例,
  每个绑不同的外部连接(`s3-prod-a` / `s3-prod-b` / …);
  授权仍然在各自服务器的「能力」Tab → 能力「管理」抽屉里按能力授权
- 不要试图做"一个 storage-s3 MCP Server 通过工具入参切换 bucket"
  ——这条路被显式禁止,且工具入参不可传 `connection_id`

**停用 / 删除**

- 停用:`enabled=false`,后续运行时拿连接会报 `MCP_CONNECTION_DISABLED`
- 删除:软删除(`deleted_at` 写入),已引用的 MCP Server 调用会失败,
  迁移前请先解绑

**审计**

每次运行时解密密钥都会写一条 `mcp_external_connection.read_secret`
审计日志,可在 **审计日志** 页过滤 resource_type 查看。

### 10.1b 如何对外提供 MCP 服务（开发方）

前面讲的是"**接入**别人的 MCP 服务"。反过来，如果你想把**自己的数据 / 能力
做成 MCP 服务**给本平台（或其它 MCP 客户端）用，有三条路，按"谁来托管运行"
区分：

| 方式 | 适合谁 | 怎么提供 | 平台侧怎么接 |
|---|---|---|---|
| **① 内置 Python server** | 平台研发，深度集成的能力 | 在 `backend/app/agent/mcp/servers/` 写一个 FastMCP server | 「预置」里加模板，`${VENV_PYTHON} -m` 启动 |
| **② 独立 stdio 包** | 想发布成可复用 npm / uvx 包 | 打成 `npx` / `uvx` 可拉起的 MCP 包 | 新增服务器，连接方式 `stdio`，启动命令填 `npx -y <你的包>` |
| **③ 远程 HTTP 服务** | 已有独立部署、不想托管到本平台 | 在你自己的服务里暴露一个 MCP Streamable HTTP 端点 | 新增服务器，连接方式 `http`，填端点 URL（见上「接入远程 HTTP MCP 服务」）|

**不管哪条路，提供方都要满足**（这样平台才能正常同步、调用）：

- 实现标准 MCP 协议的 `tools/list`（以及按需 `resources/list` / `prompts/list`），
  让平台「同步能力」能枚举出来。
- 每个工具有**清晰的入参 JSON Schema 和说明**——它会原样变成平台里的能力描述，
  Agent 靠这个决定怎么调。
- 错误**结构化返回**，不要把异常堆栈直接抛给调用方。
- **凭证不要写死在代码里**：从环境变量读，由平台通过 `env_config` / 外部连接注入
  （见 §10.1a 与技术标准 §17 安全标准）。

**开发规范（写 ① 内置 server 时必看）**：FastMCP 骨架、命名、`${VENV_PYTHON}`
启动约定、PRESETS 模板、smoke 测试等完整规范见
**《AIOS 综合技术标准》§十六 内置 MCP Server 开发规范**。本节只做选型导航，
不重复代码细节。

> 一句话选型：**深度内置选 ①，要复用 / 开源选 ②，已有独立服务选 ③。** 三种
> 提供出来后，平台侧都回到上面《快速上手》的"测试连接 → 同步能力 → 授权"主线。

| 提示词 | Agent 执行的指令 |
| 启用状态 | 开启 / 关闭 |

**常用 Cron 表达式**（时间以系统配置的 `APP_TIMEZONE` 为准，默认 Asia/Shanghai）：

| 表达式 | 频率（本地时间） |
|--------|------|
| `* * * * *` | 每分钟 |
| `*/5 * * * *` | 每5分钟 |
| `0 9 * * *` | 每天上午 9:00（本地时间） |
| `0 9 * * 1-5` | 工作日上午 9:00（本地时间） |
| `0 0 1 * *` | 每月1日零点（本地时间） |

> **时区说明**：Cron 时间以系统设置中的 `APP_TIMEZONE` 为准（系统管理 → 系统设置中可修改）。  
> 默认值为 `Asia/Shanghai`（CST, UTC+8）。如已改为 `Asia/Tokyo`，则填写 `0 9 * * *` 即表示东京时间 9:00 触发，**无需换算 UTC**。

支持手动立即触发执行（点击「▶」按钮）。

### 13.2 可靠性机制

**并发限制**：系统最多同时执行 4 个心跳任务（可通过 `HEARTBEAT_MAX_WORKERS` 调整）。

**失败自动重试**：失败后自动重试最多 2 次（30s / 60s 间隔），3 次均失败后标记 `failed`。

**漏跑补偿**：服务器重启期间漏跑的任务会在恢复后自动补执行一次（标记 `recovered`）。

**连续失败告警**：`consecutive_failures` 达到 3 次时，自动发送告警邮件到 `ALERT_EMAIL_TO`。

**Cron 表达式校验**：保存时自动验证，不合法时显示错误原因并自动禁用任务。

### 13.3 状态说明

| 状态值 | 说明 |
|--------|------|
| `completed` | 正常完成 |
| `failed` | 重试耗尽后仍失败 |
| `recovered` | 漏跑后补跑成功 |
| `cron_error` | Cron 表达式无效，任务已自动禁用 |

## 附录：常见问题

**Q：时间显示的是什么时区？**  
A：顶部导航栏的城市时钟显示东京时间（JST, UTC+9）。定时任务的 Cron 表达式以**系统配置的 `APP_TIMEZONE`** 为准（默认 `Asia/Shanghai`，可在系统管理 → 系统设置中修改）。修改后直接按本地时间填写即可，无需手动换算 UTC。

A：如果今日还没有 Agent 执行任务或调用 LLM，费用为零是正常的。

**Q：审批中心菜单上的数字是什么？**  
A：表示当前待处理的审批请求数量（24小时内创建且未处理）。

## 21. System Prompt 智能管理

> **v1.2.6 新增功能** — 适用于所有 Agent 管理员和提示词作者

本章介绍 AIOS v1.2.6 新增的 System Prompt（专属提示词）智能管理功能，帮助你在有限的 LLM 上下文窗口内写出高质量、高效率的 Agent 提示词。

---

### 21.1 为什么 System Prompt 的大小很重要？

本地部署的 LLM（如 Qwen2.5-14B，上下文窗口 8,192 tokens）对上下文长度高度敏感。一个 Agent 的上下文由多个组件叠加：

```
系统提示词（~1,595 tok）
+ Agent 专属提示词（≤ 600 tok）
+ 工具 Schema（≤ 4,000 tok）
+ 工作区注入（~699 tok）
+ Auto-Skill（≤ 600 tok）
+ 对话历史 + 用户消息 + 工具结果
─────────────────────────────────
合计上限：8,192 tokens
```

**专属提示词的上限是 600 tokens**。超出后，对话历史或工具结果会被截断，Agent 行为将变得不稳定。

---

### 21.2 CJK 文字的 Token 成本

中文和日文比英文消耗更多 Token（基于 Qwen2.5-14B 实测）：

| 语言 | 字符/Token | 相对英文倍数 |
|------|-----------|------------|
| 英文 | ~4.5 chars/tok | 1× 基准 |
| 中文 | ~1.7 chars/tok | **~2.6×** |
| 日文 | ~1.5 chars/tok | **~3×** |

**实际影响**：同样语义的提示词，中文版比英文版多消耗约 2–3 倍 Token。因此，**推荐用英文撰写专属提示词**，在 `## 工作语言` 段声明输出语言即可。

---

### 21.3 Token 色带指示器

打开任意 Agent 的编辑弹窗 → **🧬 专属 Soul** Tab，可以看到提示词编辑区顶部的色带：

```
┌─────────────────────────────────────────────────────────────┐
│ ████████████░░░░░░░░  🟡 适中 / Moderate · Custom Prompt    │
│ 320/600 tok             EN equiv ~1440 chars   [⚡ 优化]    │
└─────────────────────────────────────────────────────────────┘
│  （提示词编辑区）                                              │
└─────────────────────────────────────────────────────────────┘
```

**四级状态：**

| 色带颜色 | 状态 | Token 范围 | 含义 |
|---------|------|-----------|------|
| 🟢 绿色 | 精简 / Optimal | ≤ 200 tok | 余量充足，推荐状态 |
| 🟡 黄色 | 适中 / Moderate | 201–400 tok | 正常，在预算内 |
| 🟠 橙色 | 偏大 / Tight | 401–550 tok | 接近上限，建议压缩 |
| 🔴 红色 | 超标 / Critical | > 550 tok | 须立即裁剪 |

**统计行说明：**

- `320/600 tok`：当前估算 Token 数 / 上限
- `EN equiv ~1440 chars`：如果用英文写同等内容，需要约 1440 个字符（帮助感知 CJK 额外成本）
- 进度条颜色随 Token 占比从绿渐变至红

> **注意**：Token 数为前端启发式估算（误差 ±15%），精确值以 llama-server `/tokenize` 端点为准。

### 21.4 一键优化（压缩 + 英文翻译）

当提示词中 CJK 字符占比 > 15% 时，色带右侧出现「**⚡ 优化 / Optimize**」按钮。

**操作流程：**

1. 在编辑区写完中文 / 日文提示词
2. 点击「⚡ 优化 / Optimize」
3. 系统调用 LLM 自动完成：
   - **压缩**：去除冗余表述、合并重复规则
   - **翻译**：转换为简洁英文，保留所有约束和输出格式
4. 弹出**双栏对比 Modal**：

```
┌──────────────────────┬──────────────────────┐
│  原版 / Original     │  优化版 ✨             │
│  🔴 624 tok          │  🟢 280 tok           │
│  红色背景（原文）      │  绿色背景（英文）      │
└──────────────────────┴──────────────────────┘
          节省 344 tok（-55%）
   [取消]              [采用优化版本 / Adopt ✨]
```

5. 点击「**采用优化版本**」：编辑区内容被替换，色带立即刷新

> **注意**：优化后的英文版保留所有业务规则和输出格式要求。建议采用后检查一遍关键约束是否完整。

### 21.5 提示词撰写最佳实践

#### 推荐结构（五段式，≤ 400 tokens）

```markdown
## Role
<Agent 身份和职责范围，1–2 句>
Out of scope: <不负责的事项，一行>

## Rules
1. <最重要的约束>
2. <次要约束>
...（不超过 6 条）

## Output
1. **<段落名>** — <格式说明>
...（不超过 5 段）

## Follow-ups
<二次问答语义映射，简短>

## Style
<语言风格，1–2 句>
```

#### Token 预算建议

| 内容类型 | 推荐 Token 数 |
|---------|-------------|
| Role（角色定位） | ≤ 30 tok |
| Rules（规则列表） | ≤ 150 tok |
| Output（输出格式） | ≤ 120 tok |
| Follow-ups（二次问答） | ≤ 60 tok |
| Style（风格）| ≤ 30 tok |
| **合计** | **≤ 390 tok 🟡** |

#### 常见压缩技巧

| 原写法 | 优化写法 | 节省 |
|--------|---------|------|
| 详细说明输出表格的每一列含义 | 直接列出列名，省略解释 | ~40 tok |
| 4 条示例（"如：..."） | 1 条示例 + "依此类推" | ~30 tok |
| 重复强调"禁止" | 合并为一条 Forbidden 规则 | ~20 tok |
| 长中文句子 | 英文关键词 + 必要中文 | ~50% |

### 21.6 工具与 Skills 的 Token 影响

专属提示词只是 Token 预算的一部分。以下组件同样消耗上下文空间：

| 组件 | 上限 | 注意事项 |
|------|------|---------|
| 工具 Schema（allowed_tools） | ≤ 4,000 tok | 重型工具（email_read=956 tok）每 Agent 限 3 个 |
| Auto-Skill workflow_md | ≤ 600 tok（3×800字符） | CJK workflow_md 建议写英文 |
| 对话历史 | ≤ 400 tok（6000字符） | 长中日文对话消耗更大 |

运行 `python scripts/token_analysis.py` 可查看 Agent 全局 Token 预算状态（🟢/🟡/🔴）。

### 24.1 功能概述

### 24.2 管理 API Keys

进入路径：**系统管理 → 开放 API Keys**

| 操作 | 说明 |
|------|------|
| 创建 | 填写名称、QPS 限制、月度 Token 配额、Webhook URL（可选）、有效期（可选） |
| 查看 | 列表显示前缀（`goku_XXXXXXXX••••••••`）、用量进度、最后使用时间 |
| 编辑 | 修改限速、配额、Webhook URL |
| 吊销 | 立即失效，不可撤销 |
| 启用/禁用 | Toggle 开关暂停/恢复 Key 使用 |

> ⚠️ **API Key 仅在创建时完整显示一次**，请立即复制保存。

### 24.3 API 接入方式

所有请求需携带 Bearer Token：

```http
Authorization: Bearer goku_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

#### 提交任务

```http
POST /api/external/v1/chat
Content-Type: application/json

{
  "message": "分析上季度营收数据，生成摘要报告",
  "agent_slug": "analyst"
}
```

响应：
```json
{ "task_id": "abc123", "status": "pending" }
```

#### 轮询结果

```http
GET /api/external/v1/tasks/{task_id}
```

响应：
```json
{
  "task_id": "abc123",
  "status": "completed",
  "result": "Q4 营收同比增长 23%...",
  "error": null,
  "is_zombie": false
}
```

#### SSE 流式订阅

```http
GET /api/external/v1/tasks/{task_id}/events
Accept: text/event-stream
```

事件格式与 Web 端 SSE 相同（`card_push`、`task_completed` 等）。

### 24.4 Webhook 回调

创建 Key 时填写 Webhook URL，任务完成/失败后系统自动 POST：

```json
{
  "task_id": "abc123",
  "status": "completed",
  "result": "...",
  "error": null,
  "updated_at": "2026-05-19T10:30:00Z"
}
```

### 24.5 限速与配额

| 参数 | 说明 | 默认值 |
|------|------|--------|
| QPS | 每秒请求数上限 | 10 |
| 月度配额 | 月 Token 用量上限（0=不限） | 100,000 |
| 有效期 | Key 自动过期天数（空=永不过期） | — |

超出配额返回 `429 Too Many Requests`。

### 🕐 定时任务时区统一（v1.9.2 补丁）

**背景**：此前定时任务引擎使用服务器 OS 时区计算触发时间，与系统设置中的 `APP_TIMEZONE` 相互独立，形成"两套时间系统"。

**修复后**：心跳调度引擎（heartbeat engine）现在优先读取数据库中的 `SystemConfig.APP_TIMEZONE`，再回落到同名环境变量，最后默认 `Asia/Shanghai`。Cron 表达式的时间含义与页面上显示的系统时区完全一致。

**用户影响**：
- 若系统设置中的 `APP_TIMEZONE` 与服务器 OS 时区相同（常见情况），触发时间无变化。
- 若两者不同，任务触发时间将以 `APP_TIMEZONE` 为准（即系统设置页面中显示的时区）。
- **无需修改已有 Cron 表达式**——只要你填写的是期望的本地时间，行为即与预期一致。

*如需技术支持，请联系系统管理员或查阅项目 README 文档。*

## 32. 邮件收件箱队列管理（Email Queue）

路径：左侧菜单 → **邮件收件箱队列**（`/email-queue`）

> v1.9.3 新增。供运营人员集中查看所有来信状态、手动干预分配和派发，是 Agent 自动处理流程的管理后台。

| 卡片 | 说明 |
|------|------|

| ⚙️ 处理中 | Agent 正在处理的邮件数量 |
| 📝 草稿待审 | Agent 已起草回复，等待人工审批 |
| ✅ 已发送 | 审批通过并成功发送的邮件数量 |

### 32.2 标签页过滤

| 标签 | 显示内容 |
|------|---------|

| 新邮件 | `status = new` |
| 处理中 | `status = processing` |
| 草稿待审 | `status = draft_ready` |
| 已发送 | `status = sent` |
| 已拒绝 | `status = rejected` |

> **说明**：切换到「已拒绝」标签可查看所有被丢弃的邮件；在「全部」标签下不显示已拒绝邮件。

| 时间 | 邮件接收时间（本地时区） |
| 分配给 | 当前分配的 Agent（下拉选择器） |
| 状态 | 颜色徽章，与标签页一致 |
| 操作 | 「查看详情」「丢弃」按钮 |

### 32.4 手动分配并派发

当自动路由规则未能匹配某封邮件，或需要临时变更负责 Agent 时，可手动操作：

1. 在「分配给」列点击下拉选择器，选择目标 Agent
2. 系统弹出确认对话框：

   ```
   立即派发给 Agent 处理？
   将邮件「主题」分配给 [Agent 名称] 并立即开始处理。
   
   [立即派发]   [仅分配，不派发]
   ```

3. 选择操作：
   - **立即派发**：保存分配关系，同时创建任务交给 Agent 立即处理，邮件状态变为 `processing`
   - **仅分配，不派发**：仅更新 `assigned_agent` 字段，等待心跳任务在下次巡检时自动触发

### 32.5 查看邮件详情

点击邮件行或「查看详情」按钮，侧边抽屉展示：

| 区域 | 内容 |
|------|------|
| 基本信息 | 主题、发件人、收件地址、接收时间、状态 |
| 分配信息 | 当前 Agent、关联任务 ID（可跳转） |
| 邮件正文 | 纯文本 + HTML 双视图切换 |
| 草稿摘要 | 若已起草回复，显示草稿主题和一句话摘要 |

### 32.6 丢弃邮件

点击「丢弃」按钮，邮件状态变为 `rejected`：

- 邮件从「全部」标签消失，在「已拒绝」标签仍可查看
- 可在「已拒绝」标签中找到该邮件，通过重新分配恢复处理
- 不会触发通知或回复

### 32.7 侧边栏徽章

左侧导航栏「邮件收件箱队列」菜单项右侧显示 `new` 状态邮件数量徽章（每 30 秒自动刷新），提醒运营人员有未处理来信。

### 32.8 与审批中心的配合

典型处理闭环：

```
新邮件到达
  ↓ 心跳自动触发 / 运营手动派发（§32.4）
Agent 处理中（processing）
  ↓ Agent 起草回复 → submit_email_draft
草稿待审（draft_ready）
  ↓ 运营在审批中心（§8）查看草稿 → 可编辑 → 批准并发送
已发送（sent）
```

如 Agent 处理出错，邮件会进入 `error` 状态，运营可在邮件详情中查看错误信息，并通过重新派发恢复处理。
